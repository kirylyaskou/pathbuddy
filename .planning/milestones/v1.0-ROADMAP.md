### Phase 3: Tauri + SQLite Foundation

**Goal:** Create Tauri 2 desktop app skeleton with Vue 3 frontend, embedded SQLite via tauri-plugin-sql, and Drizzle ORM via sqlite-proxy

**Requirements:**
- TAURI-01: App runs as Tauri 2 desktop application with Vue 3 frontend
- TAURI-02: Embedded SQLite database via tauri-plugin-sql
- TAURI-03: Drizzle ORM in sqlite-proxy mode for type-safe DB access
- TAURI-04: Database migrations run on app startup

**Success Criteria:**
1. App launches as Tauri 2 desktop window with existing Vue 3 combat tracker
2. SQLite database created on first launch (pf2e.db)
3. Drizzle schema defines pf2e_entities, sync_state, and campaigns tables
4. Migrations execute on startup creating all tables and indexes

**Key Decisions:**
- Stack: Tauri 2 + Vue 3 + Pinia (existing) + SQLite + Drizzle sqlite-proxy
- Tauri plugins: tauri-plugin-sql (sqlite), tauri-plugin-http, tauri-plugin-fs
- Rust deps: reqwest (HTTP), zip (archive extraction)
- Drizzle generates SQL on frontend, sends via IPC to Rust backend
- Schema: pf2e_entities (source_id, pack, slug, name, entity_type, raw_data, content_hash, synced_at), sync_state (last_release, last_sync_at, total_entities), campaigns

### Phase 4: PF2e Data Sync

**Goal:** Implement data sync from foundryvtt/pf2e GitHub releases — download ZIP, extract, import into SQLite

**Requirements:**
- SYNC-01: User can sync PF2e data from foundryvtt/pf2e GitHub releases
- SYNC-02: System downloads release ZIP without requiring git on user machine
- SYNC-03: System upserts entities with content-hash change detection
- SYNC-04: System removes entities deleted from upstream

**Success Criteria:**
1. Sync checks latest release via GitHub API, skips if already synced
2. Downloads zipball_url (~50-80 MB) to temp directory
3. Walks packs/pf2e/**/*.json, skips _folders.json, upserts 28K+ entities
4. Deletes entities no longer in upstream, updates sync_state

**Key Decisions:**
- Data source: https://github.com/foundryvtt/pf2e releases (zipball_url)
- No git dependency — pure HTTP download
- UPSERT with ON CONFLICT(pack, slug) using content_hash for change detection
- Batch processing in groups of 500
- Rust commands: download_file, extract_zip, glob_json_files, read_text_file

**Plans:** 2/2 plans complete
- [ ] 04-01-PLAN.md — Rust I/O commands (download, extract, glob, read, cleanup) + Tauri invoke_handler registration + path:allow-temp-dir capability
- [ ] 04-02-PLAN.md — TypeScript sync service with full pipeline (release check, ZIP download, entity import, content-hash upsert, deletion, sync_state update) + unit tests

### Phase 5: Cross-Reference System

**Goal:** Slug-based resolution of embedded creature items to canonical entities in the database, plus a creature detail slide-over panel showing linked canonical spells/items and NPC-unique abilities

**Requirements:**
- XREF-01: System resolves embedded creature items to canonical entities via slug
- XREF-02: Creature detail view shows linked canonical spells/items
- XREF-03: NPC-unique abilities render from embedded data

**Success Criteria:**
1. resolveCreatureItems() maps embedded items to canonical entities via system.slug
2. Items with matching canonical show link to full entity description
3. NPC-unique items (no canonical match) render from embedded data directly

**Key Decisions:**
- Linking key: system.slug on embedded items matches filename (slug) of canonical entity
- Type-aware matching: slug + entityType composite key prevents cross-type collisions
- Stats: 56% of embedded items have system.slug, 87% of those match a canonical
- Single batch query (inArray) instead of N+1 per item
- Canonical entities in packs have system.slug = null; slug IS the filename
- Slide-over panel (not full page route) keeps combat tracker visible
- Canonical items: blue border + link icon; NPC-unique: amber border, no link icon

**Plans:** 2/2 plans complete
- [ ] 05-01-PLAN.md — Creature type extension + resolver with TDD + Pinia creatureDetail store with TDD
- [ ] 05-02-PLAN.md — CreatureDetailPanel slide-over component + CreatureCard/CombatTracker wiring + visual verification

**Reference:** plans/plan.txt Phase 3

### Phase 6: Description Sanitization

**Goal:** Parse Foundry-specific HTML syntax (@UUID, @Damage, @Check, @Template) into clean readable HTML with internal entity links

**Requirements:**
- DESC-01: System parses Foundry @UUID links into internal entity links
- DESC-02: System renders @Damage, @Check, @Template as readable text

**Success Criteria:**
1. @UUID[Compendium.pf2e.PACK.Item.ID]{Label} renders as clickable entity link
2. @Damage[2d6[fire]] renders as "2d6 fire" in styled span
3. @Check[reflex|dc:25|basic] renders as "basic reflex DC 25"
4. @Template[cone|distance:60] renders as "cone 60 ft."

**Key Decisions:**
- Regex-based parser (no AST needed for Foundry's simple @-syntax)
- Links use data-entity-pack and data-entity-id attributes for click handling
- Graceful fallback for unrecognized @-syntax

**Plans:** 2/2 plans complete

- [ ] 06-02-PLAN.md — Wire sanitizer into CreatureDetailPanel getDescription() + click event delegation for @UUID entity links
