## Phase Details

### Phase 01: XP & Encounter Budget
**Goal**: Developers and the app can calculate encounter XP and threat ratings from party composition and creature/hazard lists
**Depends on**: Nothing (standalone module)
**Requirements**: XP-01, XP-02, XP-03, XP-04, XP-05, XP-06
**Success Criteria** (what must be TRUE):
  1. XP lookup tables return correct XP values for all level differences (-4 to +4) for creatures and both hazard types
  2. calculateCreatureXP and getHazardXp return correct values matching the PF2e CRB tables for party vs target level differences
  3. generateEncounterBudgets returns the five threat thresholds (trivial/low/moderate/severe/extreme) correctly scaled by party size
  4. calculateEncounterRating maps a total XP value to the correct threat rating string
  5. calculateXP orchestrates the full pipeline and returns correct total XP and threat rating for a mixed creatures + hazards encounter
**Plans**: 2 plans

Plans:
- [x] 01-01: XP lookup tables and creature/hazard XP calculation functions
- [x] 01-02: Encounter budget thresholds, threat rating, and orchestration function

### Phase 02: Damage Constants & Types
**Goal**: The codebase has a single authoritative source for all PF2e damage type constants, die size constants, and TypeScript interfaces that all downstream modules import
**Depends on**: Nothing (standalone foundation)
**Requirements**: DMG-01, DMG-02, DMG-03
**Success Criteria** (what must be TRUE):
  1. All physical, energy, and other damage type constants are exported with their category mappings (physical/energy/other)
  2. Material damage effects, die size progression (d4-d6-d8-d10-d12), and die face constants are exported as typed values
  3. TypeScript interfaces DamageFormula, BaseDamage, IWRBypass, and CriticalInclusion are defined and compile without errors
  4. No runtime dependency on Foundry VTT -- the module imports only from TypeScript stdlib and itself
**Plans**: 1 plan

Plans:
- [ ] 02-01: Damage constants, die sizes, material effects, and TypeScript interfaces (TDD)

### Phase 03: Modifier System
**Goal**: The app can aggregate lists of typed PF2e modifiers with correct stacking rules into a single totalModifier value
**Depends on**: Phase 02 (damage type constants/interfaces)
**Requirements**: MOD-01, MOD-02, MOD-03, MOD-04
**Success Criteria** (what must be TRUE):
  1. Modifier class instances carry slug, label, value, type (one of 7 PF2e types), and enabled state
  2. applyStackingRules returns highest bonus and lowest penalty per typed category, with all untyped bonuses and penalties stacking
  3. StatisticModifier aggregates a modifier list into a correct totalModifier using stacking rules
  4. DamageDicePF2e holds diceNumber, dieSize, damageType, category, and critical flag correctly
  5. Unit tests confirm stacking behavior: two status bonuses yield only the higher; one status + one circumstance both apply
**Plans**: 1 plan

Plans:
- [ ] 03-01: Modifier class, applyStackingRules, StatisticModifier, and DamageDicePF2e (TDD)

### Phase 04: Damage Helpers
**Goal**: Damage type categorization and die size stepping utilities are available for use by IWR and higher-level damage logic
**Depends on**: Phase 02 (damage type constants)
**Requirements**: DMG-04, DMG-05
**Success Criteria** (what must be TRUE):
  1. DamageCategorization correctly maps any damage type string to its category (physical/energy/other) and maps category names back to their member types
  2. nextDamageDieSize steps a die size up or down through the d4-d6-d8-d10-d12 progression and caps at the boundaries (no step past d4 down, no step past d12 up)
  3. Unit tests cover all valid die sizes in both directions and boundary conditions
**Plans**: 1 plan

Plans:
- [ ] 04-01: DamageCategorization utility and nextDamageDieSize function (TDD)

### Phase 05: IWR Engine
**Goal**: The app can apply a creature's immunities, weaknesses, and resistances to a damage instance and receive the correct adjusted damage value
**Depends on**: Phase 02 (damage types/interfaces), Phase 04 (damage helpers)
**Requirements**: IWR-01, IWR-02, IWR-03, IWR-04
**Success Criteria** (what must be TRUE):
  1. Immunity, Weakness, and Resistance data models accept type, value (where applicable), and exceptions arrays
  2. applyIWR applies immunities first (zeroing damage), then weaknesses (adding bonus damage), then resistances (reducing remaining damage) in the correct PF2e order
  3. Critical-hit immunity correctly un-doubles crit damage, and precision immunity reduces only the precision component to zero
  4. Weakness doubleVs (double the weakness value on a crit) and applyOnce (applies only once regardless of damage instances) special cases produce correct values
  5. Unit tests confirm a creature with fire immunity takes 0 fire, a creature with fire weakness takes extra fire, and a creature with physical resistance reduces physical correctly
**Plans**: 1 plan

Plans:
- [ ] 05-01: IWR data models, applyIWR core function, and special case handling (TDD)

### Phase 06: Conditions Module
**Goal**: The app has a complete set of PF2e condition slugs and a ConditionManager that enforces valued conditions, group exclusivity, and dying/wounded interaction
**Depends on**: Phase 02 (damage type interfaces provide naming patterns)
**Requirements**: COND-01, COND-02, COND-03
**Success Criteria** (what must be TRUE):
  1. CONDITION_SLUGS exports all 44 PF2e conditions, VALUED_CONDITIONS lists conditions that carry a numeric value, and CONDITION_GROUPS maps detection/attitude group slugs
  2. ConditionManager supports add, remove, has, and get operations and correctly tracks numeric values for valued conditions
  3. Removing dying increments wounded by 1, and adding a condition from a group removes all other conditions in that group
  4. Unit tests confirm dying removal increments wounded and group exclusivity removes the old attitude before adding the new one
**Plans**: 1 plan

Plans:
- [ ] 06-01-PLAN.md -- Condition constants, ConditionManager class, and dying/wounded/group logic (TDD)
