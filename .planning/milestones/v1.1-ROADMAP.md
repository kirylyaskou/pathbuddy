## Phase Details

### Phase 01: Bug Fix and UI Improvements

**Goal:** Fix the Tauri HTTP scope bug blocking data sync, then overhaul the entire UI with a dark fantasy design system — persistent sidebar navigation, redesigned creature cards with HP bars and condition badges, and a full-page sync view.
**Depends on:** v1.0 MVP
**Requirements:** HTTP scope fix, dark fantasy design tokens, sidebar navigation, creature card redesign with HP bar + condition badges, sync page, test suite green
**Success Criteria** (what must be TRUE):
  1. Data sync completes without HTTP permission errors
  2. App renders charcoal/gold/crimson palette with Cinzel font headings throughout
  3. Persistent 180px sidebar is visible on all pages with active-link highlighting
  4. Creature cards show HP bar and condition badge pills
  5. All 178 tests pass with no regressions
**Plans:** 4/4 plans executed -- COMPLETE

Plans:
- [x] 01-01-PLAN.md -- HTTP scope fix + design system foundation + app shell (sidebar, routing, SyncView)
- [x] 01-02-PLAN.md -- Condition value types/store + HPController amount input + ConditionBadge component
- [x] 01-03-PLAN.md -- CreatureCard + CombatTracker + AddCreatureForm dark fantasy redesign
- [x] 01-04-PLAN.md -- CreatureDetailPanel restyle + test suite updates

### Phase 02: Types, Utilities, and Query Layer

**Goal:** Establish all code contracts, data shapes, and query functions that every downstream phase depends on — with zero UI, zero layout risk, and full test coverage before any component touches these APIs.
**Depends on:** Phase 01
**Requirements:** (foundational layer — no direct v1.1 user-observable requirements; enables COMP-02–07, WORK-02, WORK-04)
**Success Criteria** (what must be TRUE):
  1. `getHpAdjustment(tier, level)` returns the correct delta for all 12 PF2e bracket boundaries (verified by unit tests at each boundary)
  2. `filterEntities(filters)` returns correct results for each filter axis (type, level range, rarity, name FTS, family, tags) verified against the production 28K database
  3. Migration v3 runs cleanly on a fresh database and adds the STORED `family` column with an index
  4. The `/compendium` route is registered in vue-router and navigates without error
**Plans:** 3/3 plans complete

Plans:
- [x] 02-01-PLAN.md -- WeakEliteTier type + weak/elite HP adjustment utility (TDD)
- [x] 02-02-PLAN.md -- Migration v3 (family column) + Drizzle schema + CompendiumView stub + route
- [x] 02-03-PLAN.md -- filterEntities query function with EntityFilter/EntityResult types (TDD)

### Phase 04: Compendium Page

**Goal:** Users can open the Compendium from the sidebar and browse, filter, and inspect all 28K synced PF2e entities — the complete Compendium feature end-to-end.
**Depends on:** Phase 03
**Requirements:** COMP-01, COMP-02, COMP-03, COMP-04, COMP-05, COMP-08
**Success Criteria** (what must be TRUE):
  1. User can navigate to the Compendium via the sidebar and see entities grouped by type (creatures, spells, hazards, items, feats, etc.)
  2. User can filter entities by type, level range (1-20), and rarity (common/uncommon/rare/unique) and see results update
  3. User can type a name into the search field and see FTS-matched entities appear within 300ms of stopping typing
  4. User can click any entity row and see its full stat block in a slide-over panel without leaving the Compendium

### Phase 05: 3-Panel Combat Workspace

**Goal:** The Combat Tracker page becomes a 3-panel workspace — creature browser on the left, tracker in the center — where users can search for creatures, choose quantity and weak/elite tier, add them to combat, and see tier labels in the tracker.
**Depends on:** Phase 04
**Requirements:** WORK-01, WORK-02, WORK-03, WORK-04, WORK-05
**Success Criteria** (what must be TRUE):
  1. The Combat page renders three visible columns — creature browser (left), combat tracker (center), detail panel placeholder (right) — filling the viewport height without collapsing
  2. User can type a name, select filters, and see matching creatures appear in the left-panel browser
  3. User can set a quantity (e.g., 3) and click Add; three auto-numbered entries ("Goblin 1", "Goblin 2", "Goblin 3") appear in the combat tracker
  4. User can choose Elite or Weak tier before adding; the creature's starting HP is adjusted per the official PF2e 12-bracket table
  5. Added Elite or Weak creatures show their tier label (e.g., "Elite: Adult Black Dragon") in the combat tracker row
**Plans:** 3/3 plans complete

Plans:
- [x] 05-01-PLAN.md -- Creature type tier field + addFromBrowser store action with auto-numbering (TDD)
- [x] 05-02-PLAN.md -- CreatureBrowser combat mode + CombatAddBar component
- [x] 05-03-PLAN.md -- CombatView 3-panel grid shell + integration wiring + tests
